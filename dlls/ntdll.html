ntdll.dll

Native API Layer - The lowest-level user-mode interface to the Windows NT kernel

Contents

Overview

ntdll.dll (NT Layer DLL) is the most fundamental Dynamic Link Library in Windows NT-based operating systems. It serves as the primary bridge between user-mode applications and the Windows NT kernel (ntoskrnl.exe). Unlike higher-level DLLs such as kernel32.dll or user32.dll, ntdll.dll exports the Native API - a collection of functions that directly invoke kernel-mode system services via the syscall instruction.

Every user-mode process in Windows automatically loads ntdll.dll during initialization. The DLL is loaded by the Process Environment Block (PEB) setup routine before the process entry point is called, making it available to all code executing in the process context.

Key Point

ntdll.dll is undocumented by Microsoft. The Native API functions are subject to change between Windows versions without notice. Production software should use the documented Win32 API instead.

Core Responsibilities

  • System call stubs for kernel-mode transitions
  • Exception dispatch and handling infrastructure
  • Loader functions for DLL initialization and management
  • User-mode APC (Asynchronous Procedure Call) queuing
  • Debug event reporting and handling
  • Hardware exception filtering
  • Heap management internals
  • String manipulation routines (Rtl* functions)

Architecture

ntdll.dll operates at the boundary between user-mode (Ring 3) and kernel-mode (Ring 0). When a Native API function is called, it prepares the system call number and arguments, then executes the syscall instruction to transition to kernel mode.

+---------------------------------------------------+ | User Mode (Ring 3) | | | | +-------------+ +--------------------------+ | | | Your Code | | kernel32.dll / Win32 | | | +-------------+ +--------------------------+ | | | | | | +----------------------+ | | | | | +-------------------+ | | | ntdll.dll | | | | (Native API) | | | +-------------------+ | | | | | SYSCALL / INT 2E | +---------------------------------------------------+ | | | | Kernel Mode (Ring 0) | | | | | +-------------------+ | | | ntoskrnl.exe | | | | (NT Kernel) | | | +-------------------+ | +---------------------------------------------------+

Calling Convention

Native API functions use the __stdcall calling convention on x86 and the Microsoft x64 calling convention on x64. Parameters are passed in registers for x64:

Register Purpose
RCX First integer/pointer argument
RDX Second integer/pointer argument
R8 Third integer/pointer argument
R9 Fourth integer/pointer argument
Stack Fifth and subsequent arguments

System Call Mechanism

On modern Windows (Vista and later), system calls use the SYSCALL instruction. The system call number is loaded into the EAX register before execution:

assembly 
; Example: NtCreateFile system call stub
mov r10, rcx          ; First parameter
mov eax, 55h          ; Syscall number - Win10 21H2, varies by build
syscall               ; Transition to kernel mode
ret                   ; Return to caller
Warning

System call numbers change between Windows versions. Hardcoding these values will cause compatibility issues. Always use the exported function names instead.

Key Exports

ntdll.dll exports over 2,000 functions. These can be categorized into several groups:

Nt* Functions - System Calls

Functions prefixed with Nt (or Zw) are system call stubs that transition to kernel mode:

  • NtCreateFile - Create or open a file
  • NtAllocateVirtualMemory - Allocate virtual memory
  • NtCreateProcess - Create a new process
  • NtCreateThread - Create a new thread
  • NtReadVirtualMemory - Read process memory
  • NtWriteVirtualMemory - Write process memory
  • NtQuerySystemInformation - Query system information

Rtl* Functions - Runtime Library

Runtime library functions that operate entirely in user mode:

  • RtlAllocateHeap - Allocate memory from heap
  • RtlCopyMemory - Memory copy operation
  • RtlEnterCriticalSection - Acquire critical section
  • RtlInitializeCriticalSection - Initialize critical section
  • RtlUnicodeStringToAnsiString - String conversion

Ldr* Functions - Loader

DLL loader functions for module management:

  • LdrLoadDll - Load a DLL into process
  • LdrGetProcedureAddress - Get function address
  • LdrUnloadDll - Unload a DLL

NtCreateFile

NtCreateFile creates or opens a file, directory, device, or other filesystem object. It is one of the most complex and frequently used Native API functions.

Function Signature

NTSTATUS NtCreateFile(
  PHANDLE            FileHandle,
  ACCESS_MASK        DesiredAccess,
  POBJECT_ATTRIBUTES ObjectAttributes,
  PIO_STATUS_BLOCK   IoStatusBlock,
  PLARGE_INTEGER     AllocationSize,
  ULONG              FileAttributes,
  ULONG              ShareAccess,
  ULONG              CreateDisposition,
  ULONG              CreateOptions,
  PVOID              EaBuffer,
  ULONG              EaLength
);

Parameters

Parameter Type Description
FileHandle PHANDLE Pointer to HANDLE that receives the file object handle
DesiredAccess ACCESS_MASK Access flags (FILE_READ_DATA, FILE_WRITE_DATA, etc.)
ObjectAttributes POBJECT_ATTRIBUTES Pointer to OBJECT_ATTRIBUTES structure
IoStatusBlock PIO_STATUS_BLOCK Pointer to IO_STATUS_BLOCK for operation result
AllocationSize PLARGE_INTEGER Optional initial file size in bytes
FileAttributes ULONG File attributes (FILE_ATTRIBUTE_NORMAL, etc.)
ShareAccess ULONG Sharing mode (FILE_SHARE_READ, FILE_SHARE_WRITE)
CreateDisposition ULONG Action if file exists/doesn't exist (FILE_OPEN, FILE_CREATE)
CreateOptions ULONG Options (FILE_SYNCHRONOUS_IO_NONALERT, etc.)
EaBuffer PVOID Optional extended attributes buffer
EaLength ULONG Length of extended attributes buffer

Return Values

Status Code Value Description
STATUS_SUCCESS 0x00000000 Operation completed successfully
STATUS_OBJECT_NAME_NOT_FOUND 0xC0000034 Specified file path does not exist
STATUS_ACCESS_DENIED 0xC0000022 Insufficient permissions
STATUS_SHARING_VIOLATION 0xC0000043 File is already opened with incompatible sharing

Example 1: Opening a File for Reading

c 
#include 
#include 

typedef NTSTATUS (NTAPI *NtCreateFile_t)(
  PHANDLE, ACCESS_MASK, POBJECT_ATTRIBUTES,
  PIO_STATUS_BLOCK, PLARGE_INTEGER, ULONG,
  ULONG, ULONG, ULONG, PVOID, ULONG
);

int main() {
  HMODULE hNtdll = GetModuleHandleA("ntdll.dll");
  NtCreateFile_t NtCreateFile = (NtCreateFile_t)
    GetProcAddress(hNtdll, "NtCreateFile");
  
  UNICODE_STRING fileName;
  RtlInitUnicodeString(&fileName, L"\\??\\C:\\test.txt");
  
  OBJECT_ATTRIBUTES objAttr;
  InitializeObjectAttributes(&objAttr, &fileName,
    OBJ_CASE_INSENSITIVE, NULL, NULL);
  
  HANDLE hFile;
  IO_STATUS_BLOCK iosb;
  
  NTSTATUS status = NtCreateFile(
    &hFile,
    FILE_READ_DATA | SYNCHRONIZE,
    &objAttr,
    &iosb,
    NULL,
    FILE_ATTRIBUTE_NORMAL,
    FILE_SHARE_READ,
    FILE_OPEN,
    FILE_NON_DIRECTORY_FILE | FILE_SYNCHRONOUS_IO_NONALERT,
    NULL,
    0
  );
  
  if (NT_SUCCESS(status)) {
    NtClose(hFile);
  }
  
  return status;
}
Note: NtClose requires a separate GetProcAddress call or typedef declaration. Add: typedef NTSTATUS (NTAPI *NtClose_t)(HANDLE);

Example 2: Creating a New File

c 
NTSTATUS CreateNewFile(PCWSTR filePath) {
  UNICODE_STRING fileName;
  RtlInitUnicodeString(&fileName, filePath);
  
  OBJECT_ATTRIBUTES objAttr;
  InitializeObjectAttributes(&objAttr, &fileName,
    OBJ_CASE_INSENSITIVE, NULL, NULL);
  
  HANDLE hFile;
  IO_STATUS_BLOCK iosb;
  LARGE_INTEGER allocSize;
  allocSize.QuadPart = 4096;
  
  NTSTATUS status = NtCreateFile(
    &hFile,
    FILE_WRITE_DATA | FILE_WRITE_ATTRIBUTES | SYNCHRONIZE,
    &objAttr,
    &iosb,
    &allocSize,
    FILE_ATTRIBUTE_NORMAL,
    0,
    FILE_CREATE,
    FILE_NON_DIRECTORY_FILE,
    NULL,
    0
  );
  
  if (status == STATUS_SUCCESS) {
    if (iosb.Information == FILE_CREATED) {
      wprintf(L"File created successfully\n");
    }
    NtClose(hFile);
  }
  
  return status;
}

Inter-Component Relationships

  • kernel32.dll CreateFileW calls NtCreateFile internally
  • I/O Manager in ntoskrnl.exe processes the request
  • File system drivers (NTFS, FAT32) handle the actual operation
  • Cache Manager may cache file data in memory

NtAllocateVirtualMemory

NtAllocateVirtualMemory allocates virtual memory within the address space of a process. This is the fundamental memory allocation primitive used by all higher-level allocation functions.

Function Signature

NTSTATUS NtAllocateVirtualMemory(
  HANDLE    ProcessHandle,
  PVOID     *BaseAddress,
  ULONG_PTR ZeroBits,
  PSIZE_T   RegionSize,
  ULONG     AllocationType,
  ULONG     Protect
);

Parameters

Parameter Type Description
ProcessHandle HANDLE Handle to target process (must have PROCESS_VM_OPERATION)
BaseAddress PVOID* Pointer to base address (NULL for system-selected)
ZeroBits ULONG_PTR Number of high-order address bits that must be zero
RegionSize PSIZE_T Pointer to size in bytes (rounded to page boundary)
AllocationType ULONG MEM_COMMIT, MEM_RESERVE, or both
Protect ULONG Page protection (PAGE_READWRITE, PAGE_EXECUTE, etc.)

Example: Allocating Executable Memory

c 
typedef NTSTATUS (NTAPI *NtAllocateVirtualMemory_t)(
  HANDLE, PVOID*, ULONG_PTR, PSIZE_T, ULONG, ULONG
);

PVOID AllocateExecutableMemory(SIZE_T size) {
  HMODULE hNtdll = GetModuleHandleA("ntdll.dll");
  NtAllocateVirtualMemory_t NtAllocateVirtualMemory =
    (NtAllocateVirtualMemory_t)GetProcAddress(hNtdll,
      "NtAllocateVirtualMemory");
  
  PVOID baseAddress = NULL;
  SIZE_T regionSize = size;
  
  NTSTATUS status = NtAllocateVirtualMemory(
    GetCurrentProcess(),
    &baseAddress,
    0,
    ®ionSize,
    MEM_COMMIT | MEM_RESERVE,
    PAGE_EXECUTE_READWRITE
  );
  
  if (!NT_SUCCESS(status)) {
    return NULL;
  }
  
  return baseAddress;
}
Tip

VirtualAlloc in kernel32.dll is a wrapper around NtAllocateVirtualMemory. For most applications, use VirtualAlloc instead of calling the Native API directly.
